开源数据库对比

总结自网络文章，点击原文查看。

共计对11种数据库进行对比，对比整理见下表

名称 特点 适用场景（优势） 不适用场景(弊端)
PostgreSQL • 特有的内置数据类型：Array、Range、UUID、Geolocation等（可以原生支持地理位置索引、范围索引等操作）
• 原生支持文档存储（JSON样式）、XML和键值存储
• 支持同步、异步复制
• 支持Python、PL、Perl等脚本语言操作
• 支持全文搜索		 1、相对其他关系数据库来说，PostgreSQL是更好的选择（比如以位置为基础的应用，需要大量数组操作的应用等）；
2、面向混合数据模型（需要部分NoSQL功能）需求时，PostgreSQL是较好的选择，因为原生支持JSON/XML/Key-Value存储。		 1、当数据模型不是关系型或者有非常特定的体系结构需求时，PostgreSQL不适用；
2、处理大型数据集时，数据库性能会急剧下降。
MariaDB • 真正意义上的开源（MySQL的原作者创建，MySQL被Oracle母公司接管）
• 面向不同需求有较多数据引擎可选（例如，用于分布式事务的Spider引擎，用于并行、分布式存储，支持海量数据仓库的ColumnStore引擎等等）
• 与MySQL相比速度有所提高，尤其是Aria引擎用于复杂查询时
• 虚拟列（存在与表中的列，值是根据确定的表达式或者根据表中其他列的值自动计算，有‘VIRTUAL和’PERSISTENT‘两种）
• 更好的复制功能（例如多源复制）
• 支持一些JSON函数
• 动态列（允许一个表中每行可以存储不同的列的集合，表的每一行都可以有一个虚拟列）		 1、寻求MySQL的替代品时（MySQL往MariaDB一般是单向迁移）；
2、针对项目种已有的关系数据模型，可以考虑选用MariaDB的新数据引擎。		 1、考虑与MySQL的兼容性，如果相关CMS（Content Management System，内容管理系统，WordPress、Joomla、 Magento等已开始支持MariaDB）不支持Maria DB，不建议更换使用MariaDB
CockroachDB • 分布式SQL数据库
• 可扩展性、强一致性和强生存性
• 数据库入口为SQL接口，集群中每一节点可充当客户端SQL网关（将SQL语句变换并执行到key-value操作）		 如果依赖SQL，并酝酿MongoDB的扩展功能，可使用CockroachDB来快速建立集群，并支持SQL查询 如果现有的RDBMS工作的很好，并且可以管理好数据剧增带来的问题，不建议使用CockroachDB这样的新产品。
如果当前系统依赖一些特殊SQL处理关键事务，考虑到SQL兼容性，不建议更换。
Neo4j • 图数据库（以图数据结构处理关联数据）
• 支持事务应用和图分析
• 数据转换能力（将大规模表格数据转换为图结构）
• 特定的查询语言（Cypher）
• 可视化和特征发现		 系统底层的数据模型是基于图结构的，则该选择Neo4j，例如社交网络分析。 底层数据不是基于图结构关系的，不需要选择Neo4j。
MongoDB • 文档型数据库（非关系数据库，BSON结构存储，获取用户对象时自动获取关联数据，没有join操作）
• 模式（schema）自由，应对特殊的/不可预测的用户需求
• 支持完全索引（可任意属性上建立索引，支持基于地理空间的索引创建）
• 支持大部分SQL查询操作
• 聚合功能丰富（如count、group、MapReduce等）
• 自动数据切片和聚合（开启配置项即可）
• 集群中添加、删除节点简单
• 针对非常快速的写入进行了优化，非常适合作为存储分析数据的缓存系统
• 支持分布式事务锁		 • 网站数据：Mongo 非常适合实时的插入，更新与查询，并具备网站实时数据存储所需的复制及高度伸缩性。
• 缓存：由于性能很高，Mongo 也适合作为信息基础设施的缓存层。在系统重启之后，由Mongo 搭建的持久化缓存层可以避免下层的数据源过载。
• 大尺寸、低价值的数据：使用传统的关系型数据库存储一些数据时可能会比较昂贵，在此之前，很多时候程序员往往会选择传统的文件进行存储。
• 高伸缩性的场景：Mongo 非常适合由数十或数百台服务器组成的数据库，Mongo 的路线图中已经包含对MapReduce 引擎的内置支持。
• 用于对象及JSON 数据的存储：Mongo 的BSON 数据格式非常适合文档化格式的存储及查询。		 • 高度事务性的系统：例如，银行或会计系统。传统的关系型数据库目前还是更适用于需要大量原子性复杂事务的应用程序。
• 传统的商业智能应用：针对特定问题的BI 数据库会产生高度优化的查询方式。对于此类应用，数据仓库可能是更合适的选择。
• 需要SQL 的问题。
RethinkDB • 实时、开源、分布式、面向文档数据库
• 变更提要（可实时地将最新数据提要推送到应用程序，克服“轮询”的局限性）
• ReQL查询语言（所有查询可链）		 • 实时性要求高的系统（如游戏） 产品商业推广基本宣告失败，公司宣布关闭，官网仍有新版本推出。
Redis • 基于内存（也可持久化）、日志型、key-value数据库
• 值类型包括：字符串、哈希、列表、集合、有序集合
• 支持pub-sub（发布-订阅）消息通信模式
• 内存淘汰机制（保持数据都是热点数据）
• 支持事务操作（有一定缺陷，如同一事务中有一条命令执行失败，余下命令仍会继续执行，原子性不能得到保证）		 • 缓存（高性能、高并发、分布式，减轻数据库的IO压力）
• 充当分布式锁		 • 成本前提下，不适宜大规模数据
• 不适宜用于冷数据（业务上要区分冷热数据）
SQLite 轻量型关系数据库（数据库的所有内容都存于.sqlite文件，无需服务器）
• 支持全事务操作
• 每张表最多可有32000列
• 支持JSON存储
• 支持子查询、全文检索等
• 单个数据库容量可达140TB
• 最大行数量可达1GB
• 比文件I/O快35%		 • 适用于中小型信息系统的Demo演示等
• 简单应用程序，不需要完整数据库		 • 未涵盖标准SQL和数据库的所有特性（缺少集群、存储过程、脚本扩展等）
• 没有客户端可连接来查询和浏览数据库
• 随着应用规模增长，性能会下降
Cassandra 分布式列存储数据库，无中心、优秀的可扩展性，更像是一堆数据节点构成的分布式网络服务，流行的分布式结构化存储方案
• 极快速的写性能（写入速度大幅度超出读）
• 高伸展性（集群中增加节点，系统复杂性不会有所增长）
• 静态列
• 高可用性（优先保证可用性和分区容错性）
• 可调节的一致性（可根据参数设置来决定读写的一致性策略）		 • 日志和分析系统（如跟踪订单状态、包裹；存储健康追踪数据；汽车的物联网数据；电子邮件等）
• 需要处理真正非常大的数据（苹果有一个Cassandra部署，处理400多PB数据；Netflix利用Cassandra每天处理一万亿个请求）		 • 不适用高读取精度系统
• 不适用于复杂数据模型（如无法满足聚合等操作）
Timescale 时间序列数据库（时间是关注的最主要轴，面向物联网发展需要）
• 基于PostgreSQL，时序优化
• 全SQL接口
• 快速插入（每秒可实现数百万次插入）和复杂查询优化
• 自动分片（按时间、空间自动分片（chunk），自动调整chunk大小）
• 支持垂直与横向扩展		 • 不支持分布式扩展，对服务器单机性能要求高
• 时序数据压缩效果不太好
CouchDB 一个开源的面向文档的数据库管理系统，具有高度可伸缩性（即使运行在容易出现故障的硬件上，集群节点的不稳定不会影响数据库），提供高可用性和高可靠性
• 离线支持
• REST API（所有数据都有一个唯一的通过HTTP暴露的URI）
• 多主同步
• 针对移动、WEB应用有多个定制版本（如PouchDB，CouchDB Lite等）
• 抗冲击性、战场测试可靠性
• 冗余数据存储的简单集群		 • 适合读多写少的应用场景
• 类lotus notes应用（企业级通讯、协同工作平台）
• 需要多版本并发控制（MVVC，Multiversion concurrency control）的场景
• 数据节点可能不在线的场景		 • 不适用于作为通用schema引擎
• 写速度慢，不适用写性能要求高的场景
